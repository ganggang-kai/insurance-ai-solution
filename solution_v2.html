一、项目概述

本项目旨在构建一套「AI外呼筛意向 → 公众号自助投保」的智能化客户转化体系。 利用AI语音机器人完成批量外呼，精准识别意向客户后，通过专属码机制将客户无缝引导至微信公众号， 在公众号内完成信息填写、健康告知、在线支付，最终实现保单生效。 全程数据埋点可追踪，AI根据客户行为实时调整引导策略，最大化提升转化率。

三、核心业务流程

全链路共7个阶段、18个关键节点，AI根据客户行为实时调整策略：

  1. 📞 接通建信

    AI外呼开场说明来意 → 建立信任 → LLM语义筛选意向。

  2. 📲 微信搜索与关注

    语音引导客户搜索公众号 → API定时查询关注状态 → 未关注则触发引导话术，提示客户完成关注。

  3. 🎯 入口行为引导

    引导客户在公众号首页找到指定入口。根据客户行为分级介入——15秒无操作提示、30秒换方式引导。

  4. 🔑 专属码交付与输入

    口播四位专属码。客户在入口页面输入专属码后自动反显预留手机号，零手动输入。

  5. 📝 信息填写与健告

    轮询确认姓名身份证填写完成 → 调用核保API判断健告结果。

  6. ⚕️ 支付与成交

    查询支付状态 → 根据支付失败原因向客户给出对应建议（如余额不足引导换卡、网络问题引导重试等）。

  7. 🏁 闭环

    投保成功 → 告知保障生效时间，流程结束。

六、方案选型与方案落地难点

📍 说明

本项目采用「流程画布 + Agentic AI 有机结合」的技术方案：开场建信与意向筛选阶段使用流程画布保证合规可控，后续公众号引导、专属码交付、投保链路指导等阶段使用 LLM回复节点 + Tool 调用实现动态交互。本章节阐述方案设计思路及实际落地中的工程难点。本项目覆盖双产品链路（0.6元短险 + 复购产品），整体落地难度大、研发成本高。

▎方案设计：流程画布 + Agentic AI 有机结合

设计思路：以流程画布为整体编排框架，将7大阶段拆分为多个子流程。开场建信阶段使用画布原生节点保证话术合规，后续阶段每个子流程内部署一个 LLM回复节点，由 LLM 自主完成当前阶段的对话目标。目标达成后触发「退出LLM」→ 画布自动跳转至下一子流程，直至通话结束。

阶段实现方式说明
接通建信 流程画布 使用普通节点（固定开场白话术）+ LLM分类节点（意向四分类：有兴趣/犹豫/拒绝/质疑）。话术预审，合规可控。
公众号引导 LLM回复节点 子流程中部署 LLM回复节点，动态引导客户搜索并关注公众号。客户完成关注后，LLM判断目标达成 → 退出LLM → 画布跳转至专属码子流程。
专属码交付 普通节点 + LLM回复节点 先调用 get_exclusive_code 获取专属码，再通过 LLM回复节点口播并确认。客户输入码后 → 退出LLM → 跳转至投保引导子流程。
投保链路引导
（0.6元 + 复购）		 LLM回复节点 + Tool 子流程中 LLM回复节点全程引导：填写被保人信息 → 点击《完善多重保障》。中途通过 query_tracking 实时查询客户页面进度，动态调整引导话术。0.6元完成后自动过渡升级页，3s后跳转复购产品子流程。
银行卡签约 LLM回复节点 + Tool 子流程中 LLM分步引导（选卡→验证码→授权），通过 query_payment 轮询签约及扣款结果。成功 → 退出LLM → 跳转升级页；失败 → LLM分析原因引导重试。
升级过渡 画布内置 3秒自动过渡，LLM简单告知当前进展。不属于独立子流程。

子流程串联机制：每个子流程的核心是 LLM回复节点，LLM 的「退出意图」设置为「当前阶段目标完成」。当 LLM 判断客户已完成当前阶段（如已关注公众号、已输入专属码、已签约成功）时，自动触发退出 → 画布根据退出结果跳转至下一个子流程。这种设计既保留了 LLM 的动态交互能力，又避免了全程依赖 LLM 导致的上下文窗口溢出和成本不可控问题。

6.1 整体落地复杂度

本项目覆盖双产品链路、约5个子流程，涉及4种节点类型（普通/跳转/LLM分类/LLM回复）的组合编排。全链路需调试的子流程跳转逻辑、LLM退出判断阈值、以及 Tool 调用时序。同时需对接客户侧4个API接口（专属码获取、页面埋点查询、核保结果查询、支付结果查询），接口响应延迟和格式差异是联调阶段最核心的工程风险。

其中投保链路引导子流程为全链路复杂度最高的环节：客户进入公众号后的每一步操作都需要AI实时感知并动态响应，LLM回复节点需在引导话术、进度查询、异常处理之间灵活切换。任何一个跳转条件或退出阈值设置不当，都会导致客户被错误分流。

6.2 多方系统对接成本

#对接系统难点说明
1 客户微信公众号 菜单配置、模板消息、关注自动回复均需客户侧操作且我方无法直接调试，任何配置变更都需沟通确认。
2 客户投保H5页面 埋点SDK的嵌入、专属码参数接收、手机号反显等功能均依赖客户侧前端开发，任何页面改动都可能影响埋点数据格式，导致LLM回复节点的进度判断逻辑失效。
3 客户API接口 专属码获取、页面埋点查询、核保结果查询、支付结果查询等至少4个接口需与客户侧逐一联调。接口响应延迟、数据格式不一致、鉴权方式差异等问题是联调阶段最常见的耗时点。
4 我司AI外呼系统 语音合成质量、ASR识别准确率、外呼并发调度、号码健康度管理、外呼时段策略等，均需根据客户白名单特征进行专项调优。
5 LLM回复节点调优 每个子流程的LLM回复节点Prompt需独立设计，退出LLM的时机判断需反复调试（过早退出则引导不充分，过晚退出则对话冗余）。预计每个子流程至少3轮Prompt迭代。

6.3 综合工程评估

维度评估
子流程数量约5个子流程（建信/公众号引导/专属码/投保引导/银行卡签约），升级过渡为画布内置
LLM回复节点每个子流程1个，共约5个，每个需独立设计Prompt和退出条件
Tool调用节点专属码获取、埋点查询、核保查询、支付查询，共4个Tool，嵌入子流程中按需调用
子流程跳转路径正负分支合计超过20条路径，需逐条验证退出条件准确性和跳转逻辑
建设期研发投入画布编排、接口联调、Prompt工程、Tool封装、子流程联调、全链路测试，预计投入1–2名研发+1名产品经理，持续5–6周
运营期持续投入 上线后需持续投入人力进行Prompt迭代优化、LLM回复效果监控、话术策略调整、外呼时段/频次策略调优。预计长期投入至少1人进行日常运营维护，按月迭代Prompt版本

⚠️ 关键风险提示

本方案高度依赖客户侧API接口的稳定性和埋点数据的准确性。客户侧接口响应需达到毫秒级，延迟过高将直接影响AI对话流畅度和LLM回复节点的实时判断效果。以上风险需在项目启动前与客户明确对齐，避免上线后因客户侧问题导致我方效果不达标。

✅ 选型结论

综合以上分析，本项目最终采用「流程画布 + Agentic AI 有机结合」方案：开场建信阶段使用流程画布保证话术合规可控；后续阶段使用 LLM回复节点 + Tool 调用的子流程模式，兼顾动态交互能力与成本控制。该方案在AI外呼引导场景下具有更高的可控性、更快启动、以及更低的前期成本。
