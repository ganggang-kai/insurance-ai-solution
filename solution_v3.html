一、项目概述

本项目旨在构建一套「AI外呼筛意向 → 公众号自助投保」的智能化客户转化体系。 利用AI语音机器人完成批量外呼，精准识别意向客户后，通过专属码机制将客户无缝引导至微信公众号， 在公众号内完成信息填写、健康告知、在线支付，最终实现保单生效。 全程数据埋点可追踪，AI根据客户行为实时调整引导策略，最大化提升转化率。

三、方案选型与方案落地难点

📍 说明

本项目采用「流程画布 + Agentic AI 有机结合」的技术方案：开场白使用画布普通节点保证合规，后续阶段以子流程方式串联，每个子流程内部署 LLM回复节点实现动态交互。API 接口分 Demo 和 POC 两个阶段逐步对接。本章节阐述方案设计思路及实际落地中的工程难点。本项目覆盖双产品链路（0.6元短险 + 复购产品），整体落地难度大、研发成本高。

▎方案设计：流程画布 + Agentic AI 有机结合

设计思路：以流程画布为整体编排框架，将全链路拆分为 9 个子流程。开场白使用画布原生节点保证话术合规，后续每个子流程内部署一个 LLM回复节点，由 LLM 自主完成当前阶段的对话目标。目标达成后触发「退出LLM」→ 画布自动跳转至下一子流程，直至通话结束。

#子流程实现方式核心动作
1开场白流程画布—普通节点固定开场话术 + LLM分类节点意向识别（有兴趣/犹豫/拒绝/质疑）
2关注公众号LLM回复节点动态引导客户搜索并关注公众号，通过对话确认关注状态
3进入专属码输入入口LLM回复节点引导客户在公众号首页找到专属码输入位置
4输入专属码普通节点 + LLM回复节点调用 get_exclusive_code 获取码 → LLM口播 → 引导输入并确认
5填写个人信息并点击按钮LLM回复节点0.6元产品：引导填写被保人信息 → 点击《完善多重保障》
6收银台输入验证码
并签约快捷支付		LLM回复节点分步引导：选卡→验证码→授权→等待自动扣款
7第二次点击投保按钮LLM回复节点复购产品：自动跳转 → 引导点击《完善多重保障》
8第二次扣款并完成升级LLM回复节点银行卡签约（或免签）→ 扣款 → 3s升级过渡
9投保完成结束挂机画布内置告知双产品保障生效时间，结束通话

子流程串联机制：每个子流程的核心是 LLM回复节点，LLM 的「退出意图」设置为「当前阶段目标完成」。当 LLM 判断客户已完成当前阶段时，自动触发退出 → 画布根据退出结果跳转至下一个子流程。这种设计既保留了 LLM 的动态交互能力，又通过子流程隔离避免了全程依赖 LLM 导致的上下文窗口溢出和成本不可控问题。

▎API 接口分阶段对接策略

为降低初期联调风险、快速验证基础链路，API 接口分两个阶段逐步对接：

阶段对接接口说明
Demo 阶段 仅 get_exclusive_code 验证基础链路：外呼→关注→输码→投保→签约。LLM回复节点通过对话确认客户进度（如"您关注了吗？""码输好了吗？"），不依赖埋点数据。
POC 投产阶段 + query_tracking + query_payment 接入埋点查询实现页面进度监控，接入支付查询实现签约结果轮询。LLM回复节点可根据实时数据动态调整引导策略，大幅提升转化效率。

Demo 阶段的已知局限：由于仅有专属码接口，LLM无法实时感知客户在页面上的操作进度（如是否已点击按钮、是否进入收银台），只能通过对话询问客户来确认。这将导致引导话术的精准度有限，部分场景需要客户主动告知当前状态。

6.1 整体落地复杂度

本项目覆盖双产品链路、9 个子流程，涉及 4 种节点类型（普通/LLM分类/LLM回复/跳转）的组合编排。Demo 阶段仅需对接 1 个 API 接口，POC 阶段扩展至 3 个接口。全链路需调试的子流程跳转逻辑、LLM退出判断阈值、以及 Tool 调用时序。

其中收银台签约子流程（#6）为全链路复杂度最高的环节：LLM需分步引导银行卡签约全流程，在 Demo 阶段仅靠对话确认进度，对 Prompt 设计质量要求极高。任何一个跳转条件或退出阈值设置不当，都会导致客户被错误分流。

6.2 多方系统对接成本

#对接系统难点说明
1客户微信公众号菜单配置、模板消息、关注自动回复均需客户侧操作且我方无法直接调试，任何配置变更都需沟通确认。
2客户投保H5页面专属码参数接收、手机号反显、投被保人页面、银行卡收银台等功能均依赖客户侧前端开发。Demo 阶段不依赖埋点，POC 阶段需客户预置埋点SDK。
3客户API接口Demo 阶段仅对接专属码获取 1 个接口；POC 阶段新增页面埋点查询和支付结果查询 2 个接口。接口响应延迟和数据格式差异是联调阶段最常见的耗时点。
4我司AI外呼系统语音合成质量、ASR识别准确率、外呼并发调度、号码健康度管理、外呼时段策略等，均需根据客户白名单特征进行专项调优。
5LLM回复节点调优9 个子流程中 7 个部署了 LLM回复节点，每个需独立设计 Prompt 和退出条件。Demo 阶段 LLM 仅靠对话判断进度，对 Prompt 的精准度和容错性要求更高。预计每个子流程至少 3 轮 Prompt 迭代。

6.3 综合工程评估

维度评估
子流程数量9 个子流程（开场白/关注公众号/进入入口/输码/填信息/签约/二次投保/二次扣款/挂机）
LLM回复节点子流程 #2–#8 各部署 1 个，共 7 个，每个需独立设计 Prompt 和退出条件
API接口（Demo）仅 1 个——专属码获取接口，LLM 通过对话确认客户进度
API接口（POC）扩展至 3 个——新增页面埋点查询、支付结果查询，LLM 可实时查询客户状态
子流程跳转路径正负分支合计超过 25 条路径，需逐条验证退出条件准确性和跳转逻辑
建设期研发投入画布编排、接口联调、Prompt工程、子流程联调、全链路测试，预计投入 1–2 名研发+1 名产品经理，持续 5–6 周
运营期持续投入上线后需持续投入人力进行 Prompt 迭代优化、LLM回复效果监控、话术策略调整。Demo→POC 过渡期需增加埋点和支付接口联调工作。预计长期投入至少 1 人进行日常运营维护

⚠️ 关键风险提示

Demo 阶段 LLM 回复节点完全依赖对话确认客户进度，无法通过 Tool 获取页面实时状态，对 Prompt 设计质量要求极高。POC 阶段接入埋点和支付接口后，客户侧接口响应需达到毫秒级，延迟过高将直接影响 LLM回复节点的实时判断效果。以上风险需在项目启动前与客户明确对齐。

✅ 方案优势总结

优势说明
阶段性交付Demo 阶段仅需 1 个 API 接口即可跑通基础链路，3–4 周快速验证；POC 阶段逐步叠加埋点和支付接口，分步迭代、风险可控。
合规可控开场白使用画布普通节点，话术预审可追溯。后续 LLM回复节点通过子流程隔离，每个节点上下文有限，降低合规风险。
灵活交互每个子流程中 LLM 可动态应对客户各种反应（不关注、不会操作、卡在签约等），无需像纯画布方案预枚举所有分支。
成本可控话术预生成 + LLM分类仅产生少量 Token 消耗，相比全 Agentic AI 方案大幅降低 TTS 和 Token 成本。
快速启动无复杂 Tool 封装需求，流程画布编排 + Prompt 设计即可上线，项目启动速度远快于全 Agentic AI 方案。
