一、项目概述

本项目旨在构建一套「画布编排筛意向 + Agentic AI自主引导投保」的混合智能转化体系。 开场白与意向筛选阶段采用流程画布实现（合规可控），意向客户确认后进入Agentic模式——由LLM回复节点自主与客户对话， 通过Tool调用完成专属码获取等关键动作，逐阶段引导客户在微信公众号内完成投保全流程。

产品定位：0.6元短期意外险（首单）+ 复购升级产品，无健康告知，仅支持银行卡签约快捷支付。

2.3 子流程内部运行机制

🔄 LLM回复节点工作模式

每个子流程内部由一个LLM回复节点驱动，运行逻辑如下：

  • 回复意图 → LLM继续与客户对话，引导完成当前阶段目标（如引导关注公众号、引导输入专属码等）
  • 退出意图 → LLM判断当前阶段目标已达成，触发退出，画布跳转至下一子流程
  • Tool调用 → 对话过程中LLM按需调用Tool（如获取专属码），将结果融入对话

三、Agentic方案详解

本项目采用「流程画布 + Agentic AI 有机结合」的技术方案，将全链路拆分为 8个子流程：开场白使用画布普通节点保证合规，后续阶段以子流程方式串联，每个子流程内部署 LLM回复节点实现动态交互，由 LLM 自主完成当前阶段的对话目标。目标达成后触发「退出LLM」→ 画布自动跳转至下一子流程，直至通话结束。

流程画布 Agentic LLM回复节点

  1. 📞 开场白 + 意向筛选 流程画布

    AI外呼开场说明来意，建立信任。LLM分类节点识别客户意向，非意向客户礼貌结束，意向客户进入Agentic阶段。

  2. 📲 引导关注公众号 LLM回复节点

    LLM自主引导客户搜索并关注公众号。根据客户反馈灵活调整话术（如客户说找不到→换关键词引导，客户说已关注→确认后推进）。

  3. 🎯 引导进入专属码输入入口并输入专属码 LLM回复节点 + Tool

    引导客户在公众号找到指定入口。LLM调用「专属码获取」Tool拿到客户专属码并口播，引导客户输入后确认手机号反显成功。

  4. 📝 填写个人信息并点击按钮 LLM回复节点

    引导客户填写姓名、身份证号并点击确认按钮。LLM根据客户反馈判断是否完成。

  5. 💳 收银台输入验证码并签约快捷支付 LLM回复节点

    引导客户在收银台页面输入银行卡验证码，完成快捷支付签约并扣款0.6元。

  6. 🔄 第二次点击投保按钮 LLM回复节点

    首单完成后引导客户点击升级产品的投保按钮，进入复购流程。

  7. 💰 第二次扣款并完成升级 LLM回复节点

    引导客户确认复购产品信息并完成第二次扣款，签约快捷支付自动扣款无需再输验证码。

  8. 🏁 投保完成 → 结束挂机 LLM回复节点

    确认投保成功，礼貌结束通话。

▎API 接口分阶段对接策略

为降低初期联调风险、快速验证基础链路，API 接口分两个阶段逐步对接：

阶段对接接口说明
Demo 阶段 仅 get_exclusive_code 验证基础链路：外呼→关注→输码→投保→签约。LLM回复节点通过对话确认客户进度（如"您关注了吗？""码输好了吗？"），不依赖埋点数据。
POC 投产阶段 + query_tracking + query_payment 接入埋点查询实现页面进度监控，接入支付查询实现签约结果轮询。LLM回复节点可根据实时数据动态调整引导策略，大幅提升转化效率。

Demo 阶段的已知局限：由于仅有专属码接口，LLM无法实时感知客户在页面上的操作进度（如是否已点击按钮、是否进入收银台），只能通过对话询问客户来确认。这将导致引导话术的精准度有限，部分场景需要客户主动告知当前状态。

3.1 整体落地复杂度

本项目覆盖双产品链路、8 个子流程，涉及 4 种节点类型（普通/LLM分类/LLM回复/跳转）的组合编排。Demo 阶段仅需对接 1 个 API 接口，POC 阶段扩展至 3 个接口。全链路需调试的子流程跳转逻辑、LLM退出判断阈值、以及 Tool 调用时序。

其中收银台签约子流程（#6）为全链路复杂度最高的环节：LLM需分步引导银行卡签约全流程，在 Demo 阶段仅靠对话确认进度，对 Prompt 设计质量要求极高。任何一个跳转条件或退出阈值设置不当，都会导致客户被错误分流。

3.2 多方系统对接成本

#对接系统难点说明
1客户微信公众号菜单配置、模板消息、关注自动回复均需客户侧操作且我方无法直接调试，任何配置变更都需沟通确认。
2客户投保H5页面专属码参数接收、手机号反显、投被保人页面、银行卡收银台等功能均依赖客户侧前端开发。Demo 阶段不依赖埋点，POC 阶段需客户预置埋点SDK。
3客户API接口Demo 阶段仅对接专属码获取 1 个接口；POC 阶段新增页面埋点查询和支付结果查询 2 个接口。接口响应延迟和数据格式差异是联调阶段最常见的耗时点。
4我司AI外呼系统语音合成质量、ASR识别准确率、外呼并发调度、号码健康度管理、外呼时段策略等，均需根据客户白名单特征进行专项调优。
5LLM回复节点调优8 个子流程中 6 个部署了 LLM回复节点，每个需独立设计 Prompt 和退出条件。Demo 阶段 LLM 仅靠对话判断进度，对 Prompt 的精准度和容错性要求更高。预计每个子流程至少 3 轮 Prompt 迭代。

3.3 综合工程评估

综合以上分析，本项目 Demo 阶段仅需对接专属码获取 1 个接口即可跑通基础链路，POC 阶段叠加埋点和支付接口增强AI感知能力。建设期预计投入 1–2 名研发 + 1 名产品经理，持续 4 周左右。上线后需持续投入至少 1 人进行 Prompt 迭代优化和日常运营维护。

⚠️ 关键风险提示

Demo 阶段 LLM 回复节点完全依赖对话确认客户进度，无法通过 Tool 获取页面实时状态，对 Prompt 设计质量要求极高。POC 阶段接入埋点和支付接口后，客户侧接口响应需达到毫秒级，延迟过高将直接影响 LLM回复节点的实时判断效果。以上风险需在项目启动前与客户明确对齐。

✅ 方案优势总结

优势说明
阶段性交付Demo 阶段仅需 1 个 API 接口即可跑通基础链路，3–4 周快速验证；POC 阶段逐步叠加埋点和支付接口，分步迭代、风险可控。
合规可控开场白使用画布普通节点，话术预审可追溯。后续 LLM回复节点通过子流程隔离，每个节点上下文有限，降低合规风险。
灵活交互每个子流程中 LLM 可动态应对客户各种反应（不关注、不会操作、卡在签约等），无需像纯画布方案预枚举所有分支。
成本可控话术预生成 + LLM分类仅产生少量 Token 消耗，相比全 Agentic AI 方案大幅降低 TTS 和 Token 成本。
快速启动无复杂 Tool 封装需求，流程画布编排 + Prompt 设计即可上线，项目启动速度远快于全 Agentic AI 方案。
